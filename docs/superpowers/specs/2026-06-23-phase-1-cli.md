# murmur — Spec: Fase 1 (CLI real + `~/.murmur/config.json`)

- **Fecha:** 2026-06-23
- **Producto / CLI:** `murmur`
- **Estado:** Aprobado, pendiente de implementación
- **Depende de:** Fase 0 (monorepo, `@murmur/shared`, paquete `murmur`)

---

## 1. Resumen

Convertir el binario `murmur` (esqueleto en Fase 0) en un CLI con comandos **reales y con
efectos**, respaldados por un `ConfigStore` que persiste la configuración del usuario en
`~/.murmur/config.json`. Esta fase NO arranca el asistente (eso es Fase 9): `start` solo valida
prerequisitos. El objetivo es la gestión de configuración y el andamiaje de comandos, dejando el
repo en verde.

## 2. Decisiones confirmadas

| Tema | Decisión |
| ---- | -------- |
| Directorio de datos | `~/.murmur/` por defecto; **inyectable** vía variable de entorno `MURMUR_HOME` (precedencia: `MURMUR_HOME` > `~/.murmur`). Esto hace los tests herméticos sin tocar el HOME real. |
| Archivo de config | `<base>/config.json`. Se crea el directorio si falta. Permisos del archivo `0600` (solo el dueño). |
| Esquema de config | `{ openaiApiKey?: string; hotkey: string; model: string; voice: string; theme: 'system'\|'dark'\|'light' }`. Defaults: `hotkey="CommandOrControl+Shift+Space"`, `model="gpt-realtime"`, `voice="alloy"`, `theme="system"`. `openaiApiKey` opcional. |
| Validación | JSON malformado o esquema inválido → `ConfigError` con mensaje claro y ruta. Campos desconocidos se ignoran con tolerancia (no rompen). |
| Firma de `run` | `run(argv: string[], deps: CliDeps): Promise<{ stdout: string; exitCode: number }>`, donde `CliDeps = { config: ConfigStore; now?: () => number }`. Pura respecto a stdout/exit (no imprime ni sale ella misma). El binario `index.ts` construye deps reales, imprime `stdout` y hace `process.exit(exitCode)`. |
| Parsing de args | Parser manual de subcomandos (sin dependencias nuevas). El conjunto de comandos es pequeño y cerrado. |
| Redacción de la key | `config` y `status` nunca imprimen la key completa: se muestra `sk-…últimos4` o `(no configurada)`. |

## 3. Comandos (contrato)

| Comando | Efecto | Salida / exitCode |
| ------- | ------ | ----------------- |
| `murmur start` | Comprueba que hay API key. | Sin key: error explicando `murmur config set-openai-key`, `exitCode=1`. Con key: mensaje "arrancaría" + `TODO(F9)`, `exitCode=0`. |
| `murmur config` | Lee config. | Imprime config con key **redactada**, `exitCode=0`. |
| `murmur config set-openai-key <key>` | Guarda `openaiApiKey`. | Key vacía → error `exitCode=1`. OK → confirma (redactada), `exitCode=0`. |
| `murmur config set-hotkey <combo>` | Guarda `hotkey`. | Combo vacío → error. OK → confirma, `exitCode=0`. |
| `murmur memory reset [--yes]` | Borra memoria local: elimina `<base>/memory.db` si existe. SQLite real llega en F6 → `TODO(F6)`. | Sin `--yes`: pide confirmación e informa cómo confirmar (no borra), `exitCode=0`. Con `--yes`: borra (o informa que no había nada), `exitCode=0`. |
| `murmur status` | Lee config. | Imprime versión, ruta de config, API key sí/no (sin revelarla), hotkey, model, voice, ruta de datos. `exitCode=0`. |
| `murmur help` / `--help` / (sin args) | — | Ayuda con los comandos reales, `exitCode=0`. |
| `murmur --version` / `-v` | — | Versión, `exitCode=0`. |
| (desconocido) | — | Mensaje "comando desconocido" + sugerencia `murmur help`, `exitCode=1`. |

## 4. Seguridad de la API key

- Se guarda solo en `<base>/config.json` con permisos `0600`. Nunca en el repo ni en logs.
- Nunca se imprime completa (redacción obligatoria en `config` y `status`).
- `.gitignore` ya excluye datos locales; `MURMUR_HOME` real está fuera del repo.

## 5. Estrategia de tests (Vitest, sin red)

Cada test usa un `baseDir` temporal (p. ej. `fs.mkdtemp`) inyectado al `ConfigStore`; jamás se
toca el `~/.murmur` real. Cobertura mínima:

1. `ConfigStore`: roundtrip `save`/`load`; defaults cuando no hay archivo; permisos `0600` del
   archivo escrito; JSON malformado → `ConfigError`.
2. `config set-openai-key` luego `config` → la key aparece **redactada**, nunca completa.
3. `config set-openai-key ""` → error, `exitCode=1`.
4. `status` con y sin key → refleja "configurada"/"no configurada" sin revelarla.
5. `start` sin key → `exitCode=1` con instrucción; con key → `exitCode=0`.
6. `memory reset` sin `--yes` no borra; con `--yes` borra el fichero si existe.
7. Comando desconocido → `exitCode=1`. `--version` → versión. help contiene los comandos.

## 6. Criterios de aceptación

1. `pnpm install && pnpm typecheck && pnpm lint && pnpm test && pnpm build` en verde.
2. `cd packages/native && cargo test` sigue verde (la fase no toca Rust).
3. Los comandos de §3 se comportan según el contrato (cubierto por tests).
4. La API key nunca se imprime completa ni se commitea: `git grep -nE "sk-[A-Za-z0-9]{6,}"` no
   encuentra keys reales.
5. TypeScript strict sin `any` no justificados; ESLint sin errores.
6. La firma `run(argv, deps)` permite tests herméticos con `baseDir` temporal.

## 7. Fuera de alcance

Arranque real del asistente (F9), SQLite real para `memory reset` (F6), UI (F2+), hotkey global
funcional (F3). Aquí solo se persiste configuración y se valida.
